---
description: Create a new BMF project from template, optionally pre-populated from a source.
---

## User Input

```text
$ARGUMENTS
```

## Overview

Create a new BMF project by copying `bmfs/template/` to a target folder. Optionally use a source (text description, URL, or file) to pre-populate entities with real content instead of template examples.

## Execution

### 1. Parse Arguments

Extract from `$ARGUMENTS`:
- `name` (required) — project name. The project is created at `bmfs/works/{name}/`.
  - `myapp` → creates `bmfs/works/myapp/`
  - Can override full path with explicit `path:` prefix: `path:links/myapp` → `bmfs/links/myapp/`
- `source` (optional) — requirements source to pre-populate the project. Can be:
  - Free-text description of the app
  - Path to a file with requirements
  - URL to a requirements document

If `name` is empty, ask the user for the project name.

Ensure `bmfs/works/` directory exists, create if needed.

### 2. Validate Path

Check that the target folder does NOT already exist:

```bash
ls bmfs/{path}
```

**If exists:** Ask user for confirmation before overwriting.

**If `links/{name}`:** The folder `bmfs/links/` must exist. If it doesn't, create it.

### 3. Copy Template

```bash
cp -r bmfs/template bmfs/{path}
```

### 4. Clean Up Template Artifacts

Remove the template README — it's a guide for template contributors, not for project users:

```bash
rm bmfs/{path}/README.md
```

Remove template example content from meta files (they'll be regenerated by dedicated commands):

```bash
rm bmfs/{path}/_epics.yaml
rm bmfs/{path}/_test-cases.yaml
rm bmfs/{path}/_comments.yaml
```

### 5. Fix Schema Paths

The schema path in YAML files depends on nesting depth. Calculate the correct relative path from `bmfs/{path}/` to `schema.json` (project root).

**Depth calculation:**
- `bmfs/myapp/` → `../schema.json` (1 level deep from bmfs/)
- `bmfs/links/myapp/` → `../../schema.json` (2 levels deep)

Update the `# yaml-language-server` directive in ALL `.yaml` files:

```bash
# For each .yaml file in the project
sed -i '' 's|$schema=../schema.json|$schema={correct_relative_path}|g' bmfs/{path}/*.yaml
```

### 6. If Source Provided — Pre-populate Project

If a `source` argument was provided, use it to populate the project with real content.

**A. Read the source:**
- If it's a file path → read the file
- If it's a URL → fetch and extract requirements
- If it's inline text → use directly

**B. Read the template README for assembly rules:**

Read `bmfs/template/README.md` — it contains the definitive guide for how to build a BMF specification:
- Section "Как собирать спецификацию" — step-by-step entity creation order
- Section "Файлы сущностей" — what goes where
- Section "Ссылки между сущностями" — how entities reference each other

**C. Read the main.yaml entity descriptions:**

Read `bmfs/{path}/main.yaml` — it describes every entity type in detail:
- Fields and their meanings
- ID format conventions
- Relationships between types

**D. Exhaustive source analysis — extraction checklist:**

Before creating any entities, analyze the source systematically.
For EVERY category below, explicitly list what was found or write "not mentioned".
This prevents silent information loss.

**D1. Modules & features inventory:**
- List ALL functional modules/sections/chapters from the source
- Each module becomes at least one epic with its own entities
- Do NOT skip modules that seem minor — if the source describes it, capture it
- IMPORTANT: Sections titled "Additional mechanics", "Additional opportunities",
  "Ideas for engagement", "Supplementary features" (or their Russian equivalents:
  «Дополнительные механики», «Дополнительные возможности», «Идеи дополнительного
  интереса») contain REAL requirements in technical specifications. They are NOT
  optional brainstorming — they describe contracted functionality. Model them as
  entities just like primary features.
- Watch for single-paragraph subsections between major sections — these are
  standalone requirements, not transitional text. Example: a one-paragraph
  "3.8 Health data collection" between multi-page sections 3.7 and 4.1.

**D2. Non-functional requirements:**
- Performance targets (response times, page load, API latency)
- Availability / SLA (uptime %, maintenance windows)
- Scalability (concurrent users, data volume, file size limits)
- Security (authentication methods, encryption, audit)
- Accessibility (screen readers, contrast, motor impairment)
- Compliance (legal frameworks, data residency, certifications)
- Testing requirements (load testing, failover)
- Backup & recovery (frequency, RPO, RTO)

**D3. External integrations:**
- Auth providers (OAuth, SSO, government portals)
- Messaging channels (Telegram, SMS, email services, push)
- Third-party APIs (maps, payments, analytics, helpdesk)
- Import/export formats (GPX, KML, CSV, Excel, PDF)
- Social media (sharing, community, bots)
- External links and contact information

**D4. Algorithms & business logic:**
- Scoring / rating formulas (calculation rules, weights, aggregations)
- Ranking algorithms (tiebreaking, regional vs global, exclusions)
- Moderation / validation rules (AI checks, content policies)
- State machines (status transitions, stage progression, admission criteria)
- Recommendation / optimization algorithms
- IMPORTANT: Backend/algorithmic requirements are often embedded WITHIN UI
  sections. When reading a section about a screen or user flow, check each
  subsection for: optimization criteria, training/feedback loops, audit trails,
  protocol/report generation, statistics collection. These are NOT UI features
  but they ARE requirements that need entities.

**D5. Detailed specifications & reference data:**
- Enum catalogs with ALL values (categories, types, statuses)
- Filter parameters with possible values and ranges
- Classification systems, taxonomies
- Phase/stage-specific requirements (what differs between stages)
- Form fields with validation rules, masks, formats
- Consent/agreement types with legal basis
- Content limits (character counts, file sizes, upload counts)
- CRITICAL: When the source lists named items (e.g. 10 thematic directions,
  7 types of tourism, POI categories per direction, filter gradations like
  distance ranges), you MUST store the COMPLETE list in entity `data` fields.
  Count items in the source, count items in the entity — numbers must match.
  Example: if source lists 10 directions by name, entity:contest:direction
  must list all 10, not say "10 тематических направлений".

**D5a. Verbatim content — proper nouns, URLs, contacts, named lists:**
- Extract ALL proper nouns: organization names, product names, system names
- Extract ALL URLs, email addresses, phone numbers, messenger handles
- Extract ALL named categories, directions, types, roles with their exact names
- Extract per-stage task compositions (what specific tasks are on each stage)
- Extract detailed classification trees (e.g. POI types per thematic direction,
  infrastructure subcategories, filter option lists with numeric ranges)
- These MUST be stored as `data` fields on relevant entities — not generalized
  into prose. Use YAML lists and maps for structured reference data.
- Example: contact links → `data.support_channels: [{type: email, value: "support@x.ru"}, ...]`
- Example: POI classification → `data.categories_by_direction: {direction_name: [cat1, cat2, ...], ...}`
- Example: filter ranges → `data.distance_ranges: [{label: "< 1 km", max_km: 1}, ...]`

**D6. Analytics & reporting:**
- Dashboard metrics (what numbers, for whom)
- Report types (dimensions, filters, groupings)
- Export formats (Excel, PDF, API)
- KPIs and indices (engagement, conversion, quality)
- Geographic/demographic breakdowns

**D7. Lifecycle & post-launch:**
- Features activated after main flow (alumni, archives, mentorship)
- Temporal features (offline mode, day/night, seasonal)
- Verification features (certificate checking, QR scanning)
- Communication about future events

**D8. Business rules, constraints and prohibitions:**

- Exclusivity rules (user can only do X OR Y, not both)
- Threshold values for progression (N tasks to advance, minimum score, age limits)
- Content limits (character counts, file sizes, max uploads per entity type)
- Temporal constraints (deadlines, time windows, mandatory ordering of operations)
- Legal/regulatory constraints that affect UX flow (blocks until signed, consent gates)
- Prohibitions — things users are NOT allowed to do ("не допускается", "запрещено",
  "невозможно без", "блокирует")
- Participation rules (who can participate in what, combination restrictions)
- These items often have NO dedicated screen/action — they are constraints ON
  existing flows. Capture them as `data` fields or `description` on the entities
  they constrain, or as dedicated `rule:*` / `constraint:*` custom entities.

**D9. Form fields and input completeness:**

- Walk through EVERY registration/input/creation form described in the source
- List ALL fields mentioned, including optional ones, checkboxes, and toggles
- Note field types, masks, and validation rules if specified
- Check for voluntary/optional data collection (health status, preferences, surveys)
- Verify consent types are exhaustive — count them in the source and compare
  to the number modeled in entities
- Check for fields that appear in enumerated lists of 10+ items — these are
  especially prone to partial capture. Count items in source, count in entity.

After completing this checklist, you have a complete inventory of what needs to be captured.

**E. Populate entities following the assembly order from README:**

1. **main.yaml** — Update the root entity with project name and description
2. **entity.yaml** — Define data models extracted from requirements
3. **context.yaml** — Set up global context (current user, computed queries)
4. **layout.yaml** — Create page layouts based on navigation structure
5. **component.yaml** — Extract reusable UI components
6. **screen.yaml** — Create screens with layout, data-binding, and components
7. **dialog.yaml** — Add dialogs (confirmations, forms, errors)
8. **action.yaml** — Define actions for user interactions
9. **event.yaml** — Add async events if real-time features exist

**F. Place checklist findings into entities:**

Review each checklist category and ensure everything found is captured:

**D1 (Modules)** → each module should map to entities in the standard files
  (screen, action, entity, dialog, event). If a module from the source has no
  corresponding entities yet, create them now.

**D2–D7 (NFRs, integrations, algorithms, specs, analytics, lifecycle)** →
  information that doesn't fit standard entity types has several options:

  **Option A — `description` fields (simplest):**
  Add detailed descriptions to the most relevant existing entities.
  For example, a rating algorithm can be described in `entity:rating:score`'s description.
  Works well for small amounts of structured text.

  **Option B — `data` fields on existing entities:**
  Add structured data to relevant entities.
  For example, `entity:map:route` can have `data.filter_params: [...]` or
  `entity:contest:stage` can have `data.admission_criteria: ...`.
  Works well for reference data tied to a specific entity.

  **Option C — Custom entity types in separate files (recommended for large volumes):**
  Create project-specific files with custom entity types.
  Examples: `req.yaml` for NFRs, `integration.yaml` for external systems,
  `algorithm.yaml` for business logic, `spec.yaml` for catalogs.
  Custom types use base schema — any `type:epic:name` ID works.
  Works well when the source has substantial content in these categories.

  **Option D — Dedicated entities within existing files:**
  Create entities like `entity:system:nfr-performance` or `entity:system:integrations`
  within entity.yaml. Less clean but keeps file count low.

  **Guidance:** Choose the option that best preserves the GRANULARITY of the source.
  Specific numbers (99.9%, ≤2.5s, 10000 users), complete enum lists, and
  algorithm formulas must be captured in structured `data` fields — not lost
  in vague prose descriptions.

**G. Section-by-section source walkthrough (CRITICAL loss-prevention step):**

This step catches items that passed through the category-based checklist.
Do NOT skip or abbreviate it.

For EACH numbered section/subsection in the source document:

1. Read the section title and ALL content (every paragraph and bullet point)
2. For each paragraph and bullet point, answer:
   - Which entity captures this? → note the entity ID
   - If no entity captures it → CREATE one now
3. Pay special attention to:
   - Single-paragraph subsections — they are full requirements, not notes
   - Bullet points describing features not yet modeled as entities
   - Constraints and prohibitions ("не допускается", "запрещено", "только",
     "невозможно без", "блокирует")
   - Backend/algorithmic details hiding within UI-focused sections
   - Items phrased as "ideas" or "additional" — treat as MANDATORY
     (see D1 guidance)
4. Verify quantitative completeness:
   - Complete field lists (are ALL form fields from the source present?)
   - All enum values (are ALL categories/types/statuses listed?)
   - Exact numeric targets (not "fast" but "≤2.5s")
   - Per-stage/per-phase differences

Build a mental (or written) coverage table as you go:

```markdown
| Source Section | Key Items | Captured In |
|---------------|-----------|-------------|
| 3.8           | Health checkbox | entity:reg:user.data.has_health_limitations |
| 8.2           | Only 1 track | entity:reg:user.description / rule entity |
| ...           | ...       | ...         |
```

Any item without a "Captured In" target is a gap — fix it before proceeding.

**G2. Factual fidelity audit (after walkthrough):**

Go back through ALL entities and verify factual completeness:

1. **Named lists**: For every enumerated list in the source (directions, categories,
   filter values, task types per stage, POI types, transport modes, etc.),
   count items in source vs items in entities. If counts differ — fix.
2. **Contact data**: Every URL, email, phone, messenger handle, bot name from
   the source must appear in an entity `data` field. Search source for `@`, `http`,
   `8 800`, `vk.com`, `.ru`, `t.me` patterns.
3. **Per-stage content**: Verify that each competition stage has its specific
   task list, admission criteria, and rules captured — not just "tasks by stages"
   generically.
4. **Detailed classifications**: If source provides subcategories (e.g. POI types
   per direction, infrastructure types, filter gradations), they must be stored
   as structured `data`, not summarized as "various categories".
5. **Form field completeness**: For each form described in source, verify that
   ALL listed fields (including optional, conditional, and voluntary ones like
   health status checkboxes) are present in entity props.

**H. Replace ALL template examples** — every file should contain only project-specific content. Do not leave template example entities.

**H2. Screen & layout analysis (CRITICAL for navigation and utility screens):**

After all domain entities are created, analyze the full set of screens and build the navigation/layout layer:

1. **Inventory all screens** — list every `screen:*` entity created so far
2. **Identify missing utility screens** that any app needs but the source may not describe explicitly:
   - **Main / Home screen** — entry point after login, dashboard or feed
   - **Navigation** — tab bar, sidebar, drawer menu (based on screen count and grouping)
   - **Settings** — if any configurable behavior exists
   - **Profile** — if user entity exists
   - **Search / Global search** — if multiple content types exist
   - **Notifications list** — if any `event:*` entities exist
   - **Error / 404 / Offline** — standard fallback screens
3. **Create layouts** in `layout.yaml`:
   - Determine navigation pattern (tabs, drawer, stack) based on screen count and hierarchy
   - Create `layout:main` with navigation structure (tabs/sections)
   - Create `layout:auth` for pre-login screens (if auth exists)
   - Create `layout:modal` for dialog/overlay screens (if needed)
4. **Assign layouts** — ensure every `screen:*` has a `layout` reference
5. **Cross-check navigation** — every screen must be reachable from at least one other screen or navigation element. No orphan screens.

If the source includes screenshots or screen mockups:
- Analyze visual structure to determine component hierarchy
- Extract navigation patterns (tab bars, headers, back buttons)
- Map visual elements to BMF components

**I. Validate the result:**

```bash
npx ts-node utils/validate-schema.ts bmfs/{path}
npx ts-node utils/check-references.ts bmfs/{path}
```

Fix any validation errors before proceeding.

**J. Generate project README.md:**

Create `bmfs/{path}/README.md` — a human-readable summary of the project's BMF specification.
This file serves as a quick reference for anyone opening the project folder.

Structure:

```markdown

{1–2 sentence project summary from main.yaml description.}

## Entity Files

| File | Type | Count | Description |
|------|------|-------|-------------|
| entity.yaml | `entity:*` | {N} | Data models |
| screen.yaml | `screen:*` | {N} | UI screens |
| action.yaml | `action:*` | {N} | Business actions |
| dialog.yaml | `dialog:*` | {N} | Modal dialogs |
| event.yaml | `event:*` | {N} | Async events |
| context.yaml | `context:*` | {N} | Global state |
| layout.yaml | `layout:*` | {N} | Page templates |
| component.yaml | `components:*` | {N} | Reusable UI components |
{If custom files were created:}
| {filename} | `{type}:*` | {N} | {brief description} |

## Entities

{For each YAML file (including custom), list all entity IDs with a one-line description.
Group by file. Use compact format:}

### {filename}

- `{entity:id}` — {description, 1 sentence}
- `{entity:id}` — {description, 1 sentence}
- ...
```

This README is generated AFTER all entities are finalized (after validation).
It reflects the actual project contents, not the template.

### 7. If No Source — Keep Template Examples

If no source was provided, keep the template examples as-is in entity files. The user will edit them manually.

Update `main.yaml` root entity with the project name:

```yaml

```

Generate a minimal `bmfs/{path}/README.md` with the project name and the standard entity files table (same structure as Step 6J, but with template entity counts).

## Behavior Rules

- Never overwrite an existing project without explicit user confirmation
- When pre-populating from source, follow the assembly order from `bmfs/template/README.md` strictly
- Always validate schema and references after pre-populating
- Use the entity type descriptions from `main.yaml` to ensure correct field usage
- Schema path must be correct relative to the project location
- Remove ALL template examples when pre-populating — no leftover `template:example:*` entities
- Keep `main.yaml` entity type descriptions intact — they serve as in-project documentation
- Always complete the extraction checklist (step D) BEFORE creating entities — never skip categories
- Preserve exact numbers, dates, enum values, and formulas from the source — do not generalize or paraphrase quantitative data
- Treat ALL content in the source as mandatory requirements unless explicitly marked optional ("рекомендуется", "по желанию", "при наличии возможности", "опционально"). Phrases like "ideas for additional interest", "additional opportunities", "additional mechanics" (and their Russian equivalents) in technical specifications are REQUIREMENTS, not suggestions — they describe contracted deliverables.
- Never skip the section-by-section walkthrough (step G) — it is the primary defense against information loss. The category-based checklist (step D) catches most items, but single-paragraph features, business constraints, and backend details embedded in UI sections are only reliably caught by walking through the source linearly.
- Never skip the factual fidelity audit (step G2) — it catches the most common failure mode: enumerated lists, contact info, and per-stage details being summarized instead of preserved verbatim.
- When the source provides a list of N named items, ALL N items must appear in entity `data` fields. Storing "10 тематических направлений" instead of listing all 10 by name is a critical data loss. Same for POI categories, filter options, contact channels, task compositions per stage.
- Contact information (URLs, emails, phones, bot handles) must be stored in entity `data` fields, not omitted. These are deliverable requirements.
- **No trivial actions.** Do NOT create actions with a single direct effect and no logic. An action must justify its existence — it should have validation, multiple effects, conditional logic, API calls, or state mutations. If the only thing an action does is one straightforward effect (navigate, set a value, emit an event), inline that effect directly where it's triggered. Bad example: `action:nav:go-competitors: { effects: [$screen.data.competitors] }` — useless wrapper, just navigate to the screen directly from the component. Actions are for **business logic**, not for wrapping single operations.
