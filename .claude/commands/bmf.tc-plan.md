---
description: Generate tc-plan.yaml — ordered TC implementation plan grouped by stages, sorted by AI based on business logic and dependencies.
---

## User Input

```text
$ARGUMENTS
```

## Overview

Analyze all test cases, their tagged entities, and stage markers to produce a prioritized implementation plan. The output is `tc-plan.yaml` — a flat ordered list of TC IDs grouped by development stage. AI determines the optimal implementation order based on:

- Business dependencies (auth before everything, data models before UI)
- Happy paths before edge cases within same domain
- Foundational TCs (referenced by many entities) before leaf TCs (referenced by none)
- Cross-epic dependencies (if TC_A's entities reference TC_B's entities → TC_B first)

## Execution

### 1. Parse Arguments

Extract from `$ARGUMENTS`:
- `folder` (required) — path to BMF project folder (e.g., `bmfs/works/myproject`)

If folder is empty, list available projects in `bmfs/` and ask user to specify.

### 2. Verify Prerequisites

Check that both exist:
- `{folder}/_test-cases.yaml`
- `{folder}/_epics.yaml`

If missing:
```
ERROR: Required files not found in {folder}
  _test-cases.yaml: {yes/no}
  _epics.yaml: {yes/no}

Run: /bmf.create-epics and /bmf.create-tcs first.
```

### 3. Read All Sources

Read and load into context:

**A. Meta files:**
- `{folder}/_test-cases.yaml` — all TC definitions with descriptions
- `{folder}/_epics.yaml` — all epic definitions

**B. All entity YAML files:**
- `{folder}/*.yaml` (excluding `_*.yaml` and `main.yaml`)
- For each entity: extract ID, tags (especially `stage:*` and `tc:*`), and `$references` to other entities

### 4. Build TC → Entities Map

For each TC (`tc:domain:scenario`), collect:
- All entities tagged with `tc:domain:scenario` (any suffix: `:start`, `:end`, or bare)
- For each entity: its `stage:*` tag and its type (screen, action, entity, dialog, event)
- All `$references` from those entities to other entities

Output an internal map:
```
tc:auth:register →
  entities: [screen:auth:register, action:auth:do-register, entity:auth:user, dialog:auth:success]
  stages on entities: [stage:mvp, stage:mvp, stage:mvp, stage:mvp]
  references OUT: [entity:auth:session, $context.current_user]
```

### 5. Determine TC Stage

For each TC, determine its development stage. Use your judgment based on:

- **Stage tags on entities** — if all entities are `stage:mvp`, the TC is clearly mvp. If mixed — consider the "latest" stage (the entity that blocks completion)
- **TC description** — edge cases and error handling are typically later stages
- **Epic context** — some epics are inherently foundational (auth) vs optional (social)

Available stages (ordered): `design`, `mvp`, `v1`, `backlog`

If entities have no stage tags — infer from context or default to `mvp`.

### 6. Determine Implementation Order

Within each stage group, sort TCs by implementation order. Consider:

**A. Dependency signal:**
- TC_A's entities reference entities tagged by TC_B → TC_B should come first
- Auth/session TCs almost always come first
- Data model TCs (entity-heavy) before UI TCs (screen-heavy)

**B. Business logic signal:**
- Happy path TCs (`*-success`, `*-complete`) before edge cases (`*-invalid`, `*-error`, `*-empty`)
- CRUD creation before CRUD read/update/delete
- Core flows before auxiliary flows within same domain

**C. Epic layering:**
- Foundational epics first (auth, core data)
- Feature epics next (main business flows)
- Enhancement epics last (notifications, social, analytics)

**D. Fan-out signal:**
- TCs whose entities are referenced by many other TCs → earlier
- TCs whose entities reference no other TCs → later (leaf nodes)

### 7. Write tc-plan.yaml

Generate `{folder}/tc-plan.yaml`:

```yaml
# tc-plan.yaml
# Implementation order — generated by bmf.tc-plan
#
# Format: stages → ordered list of TC IDs
# Order within stage = implementation priority (top = first)

mvp:
  - tc:auth:register
  - tc:auth:login-success
  - tc:catalog:browse-products
  - tc:catalog:view-product
  - tc:cart:add-item
  - tc:cart:checkout-success
  - tc:auth:login-invalid          # edge case — after happy paths
  - tc:cart:checkout-empty          # edge case

v1:
  - tc:social:add-friend
  - tc:notifications:push-settings
  - tc:auth:password-reset          # minor branch of auth — deferred to v1
```

**Rules:**
- Use YAML list format (not objects — just TC IDs as strings)
- Group by stage, stage order: `design` → `mvp` → `v1` → `backlog`
- Omit empty stages
- Add brief inline comments for non-obvious ordering decisions (especially when edge cases of early epics appear late)
- No extra fields — just stage groups with ordered TC ID lists

### 8. Present Summary

After generating, show:

```
tc-plan.yaml generated in {folder}/

  Stages:
    mvp: {N} test cases
    v1:  {N} test cases
    ...

  Total: {N} test cases planned

  Implementation order highlights:
    First: {tc:id} — {why}
    Last:  {tc:id} — {why}

  Cross-stage deferrals:
    - {tc:domain:edge-case} → moved to {stage} (edge case of {epic}, happy paths in {earlier stage})
    ...
```

Ask the user to review and adjust the order before proceeding to `/bmf.tc-implement`.

## Behavior Rules

- Never overwrite existing `tc-plan.yaml` without confirmation
- If `tc-plan.yaml` already exists, show diff of changes and ask before overwriting
- Stage determination is AI judgment — explain reasoning for non-obvious decisions in inline comments
- Every TC from `_test-cases.yaml` must appear in the plan — no orphans
- If a TC has no entities tagged (orphan TC with no tags in entity files), flag it as a warning but still include it
- Use the same language as the project for comments and summary
- The plan is a one-time generation — user will manually adjust after review
